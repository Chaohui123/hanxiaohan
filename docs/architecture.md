# ONZO 电商自动化架构设计 v2

> Ozon 俄罗斯跨境电商 AI 自动化系统
> 基准方案：DeepSeek-VL2 + K3(视觉理解) + 影刀 RPA + GLM-Image + GLM-5.2
> v2 变更：修复执行层稳定性、多店铺风控、架构分层、兜底熔断四大缺陷

---

## 一、场景清单（按 ROI 优先级排序）

### P0 — 商品上架自动化

核心链路（已补全缺失环节）：

```
1688/淘宝选品
  → 爬虫抓取（标题/价格/规格图/详情图）
  → 本地数据校验（必填类目、尺寸、重量，拦截无效商品）
  → K3 OCR 商品图（提取图中卖点文字）
  → GLM-5.2 中译俄 + 类目语义匹配 + 属性填充
  → 图片预上传 Ozon 获取 image_id（Ozon 不支持直接传图批量创建）
  → GLM-Image 生成俄文主图（可选，原图无中文卖点则跳过）
  → Ozon API v3 批量创建草稿（每批 20 个，间隔 2s，规避 429）
```

| 子场景 | 自动化方案 | 模型/工具 | 新增保障 |
|---|---|---|---|
| 1688/淘宝 → Ozon 商品搬运 | API + AI 翻译 + 自动填属性 | K3(OCR) + GLM-5.2(翻译) | 本地数据校验层 |
| 商品图俄文化 | GLM-Image 批量生成俄文版 | GLM-Image | 原图无中文则跳过 |
| 类目自动匹配 | 语义匹配 + 属性模板 | GLM-5.2(文本推理) | 必填项完整性校验 |
| 定价自动计算 | 公式引擎 + 实时汇率 | 规则引擎 + GLM-5.2 | 亏本价拦截 |
| 批量上传 | API 直传 + 模板自动更新 | Ozon API v3 | 图片预上传 + 20条/批限流 |

### P1 — 多店铺管理与风控（三层隔离）

```
账号层：独立 Client-ID + API 密钥 + 浏览器指纹 + 代理 IP
任务层：分组错峰，调价/上架/数据采集分时段执行
监控层：记录请求频率、限流次数、审核状态，异常自动暂停
```

| 子场景 | 自动化方案 | 风控规则 |
|---|---|---|
| 多店铺商品分发 | 10 店一组，组间错峰 + 随机休眠 | 单 IP 并发 ≤ 20；禁止同账号 API+RPA 双通道 |
| 价格同步调整 | 定时巡检 + 自动调价 | 调价与上架分时段执行 |
| 库存同步 | 实时库存 Webhook | 读操作无风控限制 |
| 账号健康监控 | 定时抓取 + 异常告警 | 异常账号自动暂停所有任务 |

### P2 — 比价与选品（增加 RAG）

> **注意**：本小节描述的是 Phase2/Phase3 的长期架构目标（包括 RAG/Qdrant、向量检索等）。根据仓库强制约束（参见 `.claude/rules.md`），Phase1 明确禁止在生产代码中引入 Qdrant、RAG 或复杂 Agent 框架。如需推进 Phase2/3 功能，请先提交 PR 并完成安全与成本评估。

```
每日抓取 WB/Ozon 竞品价格 → 存入 Qdrant 向量库
  → DeepSeek-VL2 检索历史价格趋势 + 当前截图
  → 输出选品建议（而非仅单次截图分析）
```

| 子场景 | 自动化方案 | 数据来源 |
|---|---|---|
| Ozon + WB 跨平台比价 | 截图→OCR→结构化比价表 + 历史趋势 | K3 + Qdrant |
| 热销品发现 | API 抓取热销榜 + 向量检索相似品 | 萌啦数据 + DeepSeek-VL2 |
| 竞品监控 | 定时快照 + diff 分析 + 异动告警 | 爬虫 + Qdrant 向量比对 |

### P3 — 客服与售后（含人工兜底）

| 子场景 | 自动化方案 | 兜底 |
|---|---|---|
| 俄语自动回复 | FAQ 匹配 + RAG 检索历史话术 + AI 生成 | 低置信度(<0.7) 推送人工看板 |
| 退货退款处理 | 状态机 + API 自动操作 | 异常状态人工介入 |
| 评价管理 | 新差评推送 + 回复模板 | 严重差评标记人工处理 |

## 三、四大缺陷修正方案

### 缺陷 1：MCP Ozon 开源库不可靠

**问题**：`@woyax/mcp-ozon-seller` 0 Star、无官方维护、无权限隔离、无限流机制。

**修正**：自研 `ozon-api-wrapper` 替代裸调开源 MCP，保留 MCP 接口协议但底层替换。

```
ozon-api-wrapper/
├── src/
│   ├── client.ts          # Ozon API v3 核心客户端
│   ├── rate-limiter.ts    # 令牌桶限流 (30次/分, 分片错峰)
│   ├── retry.ts           # 指数退避重试 (1s→2s→4s→fail)
│   ├── circuit-breaker.ts # 熔断器 (3次429/5xx → OPEN → 30s后半开)
│   ├── auth.ts            # 账号密钥隔离 + MCP 调用鉴权
│   ├── validator.ts       # 请求/响应数据校验
│   └── fallback.ts        # API 失败 → 影刀 RPA 自动切换
├── mcp/
│   └── server.ts          # MCP 协议兼容层 (AI Agent 仍然自然语言调用)
└── tests/
```

**关键行为**：
- 正常：API 调用，每次 ≤ 20 条，间隔 2s
- 连续 3 次 429/5xx：熔断器 OPEN，自动切影刀 RPA
- 30s 后半开状态：发送 1 个探测请求，成功则 CLOSE 恢复 API
- 原始 MCP 仓库仅保留作为备用参考，不直接依赖

### 缺陷 2：多店铺并发风控

**问题**：单 IP 50 店并发 → IP 熔断连坐；设备指纹同质化 → 店群判定；API+RPA 双通道 → 异常登录审查。

**修正**：三层风控隔离。

```
┌─────────────────────────────────────────────┐
│              风控控制面 (Control Plane)       │
│                                             │
│  店铺分组 (10店/组):                          │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │ Group A  │  │ Group B  │  │ Group C  │  │
│  │ 代理IP-1 │  │ 代理IP-2 │  │ 代理IP-3 │  │
│  │ 指纹-1   │  │ 指纹-2   │  │ 指纹-3   │  │
│  │ 账号 1-10│  │ 账号11-20│  │ 账号21-30│  │
│  └──────────┘  └──────────┘  └──────────┘  │
│                                             │
│  任务调度:                                    │
│  ├─ 08:00-12:00 → 上架 (Group A→B→C 错峰)   │
│  ├─ 14:00-16:00 → 调价 (Group A→B→C 错峰)   │
│  ├─ 02:00-06:00 → 数据采集 (低风控时段)      │
│  └─ 组间间隔 ≥ 30min + 随机 0-15min 抖动     │
│                                             │
│  执行通道 (同一账号严格单通道):                │
│  ├─ API 操作 → 仅通道 A                      │
│  ├─ 页面操作 → 仅通道 B                      │
│  └─ 禁止双通道并发 (数据库标记 + 锁)          │
└─────────────────────────────────────────────┘
```

**影刀角色调整**：影刀作为无状态执行器，不接管账号指纹配置；控制面统一管理指纹/代理/账号环境。

### 缺陷 3：n8n + LangChain 耦合 + SQLite 瓶颈

**问题**：n8n 原生 LangChain 节点不适合动态推理；SQLite 无分布式能力；缺少 RAG 知识库。

**修正**：架构分层解耦。

| 层 | 工具 | 职责 | 适用场景 |
|---|---|---|---|
| 确定性流水线 | n8n | 上架、库存同步、报表导出 | 固定流程，无分支决策 |
| 动态推理 | LangGraph Agent | 选品、定价、比价 | AI 自主工具调用，多步推理 |
| 数据存储 | SQLite → PostgreSQL (≥10店) | 商品/订单/价格/指标 | 单机→分布式 |
| 向量知识库 | Qdrant | 竞品历史、售后话术、选品特征 | RAG 检索增强 |

**n8n ↔ LangGraph 交互**：
- n8n 定时触发 → 调用 LangGraph Agent API → 获取决策结果 → n8n 执行下游动作
- n8n 不直接跑 AI 推理，仅做任务调度 + 结果分发

### 缺陷 4：执行层兜底逻辑缺失

**问题**：无熔断、无优先级路由、无双校验。

**修正**：完整熔断 + 路由 + 校验机制。

```
请求进入
  │
  ├─ 场景判断
  │   ├─ 批量上架/调价/查订单 → 通道 A (API)
  │   ├─ 验证码/弹窗/前端改版 → 通道 B (RPA)
  │   └─ 其他 → 通道 A 优先
  │
  ├─ 通道 A 执行
  │   ├─ 成功 → 结果写入数据库
  │   ├─ 1-2 次 429 → 指数退避重试
  │   └─ 连续 3 次 429/5xx → 熔断器 OPEN
  │       ├─ 自动切通道 B
  │       └─ 30s 后半开探测
  │
  ├─ 通道 B 执行
  │   ├─ 队列限流: 并发 ≤ 10 窗口
  │   ├─ 多余任务排队延时执行
  │   └─ 完成 → 数据双校验
  │
  └─ 双校验 (仅通道 B 结果)
      ├─ API 读取 Ozon 后台数据
      ├─ 与 RPA 提交结果比对
      └─ 不一致 → 标记异常 + 人工审核
```

## 四、分阶段落地计划

### Phase 1：单店 MVP（2-3 周）

**目标**：跑通一条"1688 商品 → Ozon 草稿箱"全链路，单店。

| 任务 | 工时 | 产出 |
|---|---|---|
| `ozon-api-wrapper` 开发（限流/重试/熔断） | 2d | 自研 API 客户端 |
| Ozon API v3 对接验证 | 0.5d | 商品创建/查询调通 |
| K3 + GLM-5.2 接入 | 1d | OCR + 翻译可用 |
| 1688 商品抓取模块 | 1d | URL→结构化数据 |
| 本地数据校验层 | 0.5d | 必填项校验 + 亏本拦截 |
| n8n 上架工作流 | 1d | 可视化编排 |
| 端到端联调（3个商品） | 1d | 全链路验证 |
| 错误处理 + 日志 + 通知 | 0.5d | 可调试 |

**Phase 1 产出**：一个 n8n workflow，输入 1688 URL，输出 Ozon 草稿箱商品。

### Phase 2：生产化（4-6 周）

**目标**：单店稳定运行 + 多店铺框架。

| 任务 | 产出 |
|---|---|
| 多店铺管理（分组隔离 + 代理 + 指纹） | 10 店分组框架 |
| 批量上架（CSV 模板 + API 分片上传） | 日上架 50+ |
| 定价引擎（汇率 + 佣金 + 物流公式） | 自动定价 |
| 库存同步（Push + 自动调库存） | 不超卖 |
| 影刀 RPA 兜底通道 | API 故障自动切换 |
| Web 管理面板（Next.js） | 可视化操作 |
| 数据双校验 | 上架结果自动比对 |

### Phase 3：智能化（8-12 周）

**目标**：AI 驱动决策，人只做审核。

| 任务 | 产出 |
|---|---|
| LangGraph Agent 服务（选品 + 定价 + 比价） | AI 决策引擎 |
| DeepSeek-VL2 接入 | 复杂推理 |
| Qdrant 向量库 + RAG | 竞品分析 + 智能客服 |
| 竞品自动监控 | 异动实时告警 |
| 智能客服（RAG + 置信度 + 人工看板） | 80% 自动回复 |
| PostgreSQL 迁移 (≥10 店时) | 多机共享数据 |
| AI 运营周报 | 自动生成报告 |
